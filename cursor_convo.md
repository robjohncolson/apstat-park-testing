import { Link } from "react-router-dom";
import styles from "./Button.module.css";
import React from "react";

type BaseProps = {
  variant?: "primary" | "secondary" | "ghost" | "danger" | "linkLight";
  className?: string;
  children: React.ReactNode;
};

type ButtonAsLink = BaseProps & {
  to: string;
  onClick?: never;
  type?: never;
};

type ButtonAsButton = BaseProps & React.ButtonHTMLAttributes<HTMLButtonElement> & {
  to?: undefined;
};

export type ButtonProps = ButtonAsLink | ButtonAsButton;

export function Button(props: ButtonProps) {
  const {
    variant = "primary",
    className = "",
    children,
    ...rest
  } = props as ButtonProps & Record<string, unknown>;

  const classes = [styles.button, styles[variant], className]
    .filter(Boolean)
    .join(" ");

  if ("to" in props && props.to) {
    // Render as router link
    return (
      <Link to={props.to} className={classes} {...(rest as any)}>
        {children}
      </Link>
    );
  }

  return (
    <button className={classes} {...(rest as React.ButtonHTMLAttributes<HTMLButtonElement>)}>
      {children}
    </button>
  );
}

Visual-regression tests are invaluable once a UI settles down, but they can feel ‚Äúin the way‚Äù while you‚Äôre still moving pieces around.  A few tactics keep them helpful (and quiet) even during active development:

1. Snapshot only what is supposed to be stable  
   ‚Ä¢ Pages that rarely change: login, 404, basic dashboard layout.  
   ‚Ä¢ Critical components in isolation (Storybook/Chromatic-style ‚Äúcomponent screenshots‚Äù).  
   ‚Ä¢ Avoid pages that intentionally morph every sprint (e.g. LessonPage while we‚Äôre still pruning CSS).  

2. Freeze the data that drives the view  
   ‚Ä¢ We already use MSW so the same mocked username and lesson data appear every run.  
   ‚Ä¢ If a date or random ID still leaks through, mask that node before taking the screenshot (`page.locator('.timestamp').evaluate(e => e.style.visibility = "hidden")`) or use Playwright‚Äôs `mask` option.

3. Keep the diff threshold tight but not absurd  
   ‚Ä¢ `0.02` (2 %) works; anything bigger usually hides real regressions.  
   ‚Ä¢ Switch to `maxDiffPixelRatio` for full-page shots, or `maxDiffPixels` for tiny components.

4. Update snapshots intentionally, not automatically  
   ‚Ä¢ When a legitimate redesign lands, run `playwright test --update-snapshots`, inspect locally, commit the new *.png files.  
   ‚Ä¢ In CI you normally run without that flag so any accidental pixel drift fails the pipeline.

5. Take screenshots at a single, deterministic viewport  
   ‚Ä¢ We fixed ours at 1280 √ó 800 so font re-flow doesn‚Äôt break baseline height.  
   ‚Ä¢ For responsive checks, test layout constraints with traditional assertions (no horizontal scroll) instead of full screenshots.

6. Keep the suite tiny  
   ‚Ä¢ Every extra baseline image adds repo weight; stick to the 2-3 pages/components that would hurt users the most if visually broken.  
   ‚Ä¢ As soon as we finish the de-globalisation and component extraction, we can expand coverage because the UI will stabilise.

What this means for us right now  

‚Ä¢ Continue refactoring freely; when we finish the CSS-cleanup/component-library phase, run one big snapshot update and treat that as the new ‚Äúvisual contract‚Äù.  
‚Ä¢ During heavy churn, it‚Äôs OK to skip or xfail the visual tests temporarily‚Äîjust bring them back before merging to main.  

So yes, visual tests can be annoying while layouts are moving, but with tight scoping and deterministic data they still catch ‚Äúoops‚Äù moments (like unintended padding/overflow) without blocking legitimate work.
